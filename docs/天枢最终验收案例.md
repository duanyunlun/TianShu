# 天枢 / TianShu 最终验收案例

> 这是天枢的最终验收门槛。
> 若本案例未通过，则即使其他单测全部通过，也应判定本轮验收不通过。
> 目的不是验证单点能力，而是验证 Agent 是否能在真实工作流中完成实际任务。

## 1. 验收定位

本案例要求天枢在真实对话中完成一项完整的软件交付任务，并正确处理以下关键能力：

- 实际需求理解
- 工具调用与连续实施
- 回合中途引导
- 回合中途中断
- 中断后继续或改向
- 代码落地
- 构建运行
- Sub-Agent：机制链路必须可用；live 场景只作为真实模型自主触发观察实验，必须记录是否观察到 `spawn_agent`
- Remote Continuity：机制链路必须可用，至少覆盖模拟移动端只读跟随、远程审批、远程中断/恢复、远程 follow-up、断线重连和重复命令幂等
- 最终交付可供人工验收的 GUI 结果

本案例的最终交付目标为：

- 在 `Test\TianShu最终验收` 下生成一个可运行的 `.NET 10 WPF` 配置 GUI 项目
- 该 GUI 的业务目标是修改和导出 Codex 对应的 `config.toml`
- 该 GUI 以 Codex 当前配置契约、官方 schema 和现行实现为准，尽量承载所有已定义且未过时的 Codex 配置项
- 最终能够导出 Codex 可迁移使用的 `config.toml`

### 1.1 覆盖边界

本案例覆盖天枢作为真实 Agent 工作流的端到端能力，包括需求理解、工具执行、中途引导、中断恢复、代码交付、构建运行、过程工件审计与最终 GUI 产物交付。

本案例不承担以下验收职责：

- VSIX 手工界面验收：需要人工操作 VSIX 时，按 VSIX 手工验收口径另行确认；本案例只要求 CLI 工件或线程日志能提供必要证据。
- 配置字段完整性的确定性证明：字段完整性应由 Codex 配置 schema / 配置解析侧单测或专项矩阵覆盖，本案例只要求 Agent 依据 Codex 当前配置契约给出可审计的主要配置域覆盖说明。
- GUI 主观质量判断：自动脚本可做 WPF 项目存在、构建和启动探针；界面观感、布局质量和人工可用性最终由人工验收。
- formal control-plane 命令矩阵：`Memory`、`Workflow`、`Collaboration`、`Participants`、`Diagnostics`、`Artifacts`、`Catalog`、`Review`、`Feedback` 等正式命令面应由对应单测、集成测试和架构锁覆盖，不纳入本次 WPF GUI 交付任务。
- 旧 agent job / 协作树 UI：旧人工/宿主管理 agent job、`send_input`、`wait_agent`、`close_agent` 等路径不作为本案例通过证据；本案例只验收当前 Kernel Runtime loop 下受治理的 `spawn_agent -> module.sub_agent/sub_agent.spawn` 能力。

### 2.2 验收前必须清理上一轮生成物

开始本案例前，必须至少清理以下内容，避免旧结果干扰：

- `Test\TianShu最终验收` 内上一轮生成的 WPF 项目

若使用配套脚本执行，本清理动作由脚本自动完成。

### 2.3 不允许触碰用户级 Codex 配置

验收过程中：

- 绝对不允许修改当前用户级 Codex 配置：`%USERPROFILE%\.codex\config.toml`
- 可以读取用户级 Codex 配置做对照或导入示例，但不能写回该文件
- 目标 WPF 程序若需要演示导出，只能导出到 `Test\TianShu最终验收` 内的产物路径，不能覆盖 `%USERPROFILE%\.codex\config.toml`
- CLI 自动化验收本身仍需要 TianShu 运行配置；若未显式传入 `-ConfigPath`，脚本应优先只读复制工作目录级 `.tianshu\tianshu.toml` 到仓库内临时文件再用于启动 TianShu，会话配置不存在时再回退到用户级 `%USERPROFILE%\.tianshu\tianshu.toml`
- 复制 TianShu 运行配置时必须同步复制同一 home 下的 `modules` 树，并在验收隔离目录内覆盖本轮 live provider matrix；只复制 `tianshu.toml` 会丢失 route set、protocol rule set 和 provider instance，不得作为有效最终验收输入
- 上述 TianShu 运行配置副本只用于驱动本轮验收会话，不是目标 WPF 程序要编辑或导出的 Codex 配置
- 每次运行最终验收都必须重新复制覆盖 `acceptance-tianshu.toml`，不能复用上一轮的旧副本

### 2.4 配置语义来源必须正确

配置 GUI 的字段来源必须以 Codex 当前配置契约、官方 schema 和现行实现为准，不能以下列内容作为主依据：

- 用户目录下现有的 `%USERPROFILE%\.codex\config.toml`
- TianShu 自身的 `tianshu.toml`
- `codex cli help`
- 过时文档

最低字段覆盖应能让人工在 GUI 中看到并导出以下 Codex 当前配置域的主要键：模型与推理、sandbox / approval、profiles、model providers、MCP servers、tools / features、通知与外部集成。若某个域在 Codex 当前配置契约中已过时、不可映射或被替代，Agent 必须在交付说明中标注依据文件与取舍原因。

### 2.5 验收架构必须对齐当前实现

本案例必须验证当前 TianShu 主链路，而不是旧 AppHost turn loop 或历史安装产物：

- CLI 自动化验收默认必须通过当前分支源码开发入口执行，即 `dotnet run --project src/Presentations/TianShu.Cli/TianShu.Cli.csproj -- chat ...`；除非另有明确发布验收要求，不得直接调用用户级 `%USERPROFILE%\.tianshu\bin\tianshu.exe` 作为本案例证据来源。
- 当前 `chat` 命令仍需要 `--apphost-project` 来解析和启动交互宿主入口。该参数只代表 Host Gateway / Control Plane 所需的宿主启动入口，不代表已删除的 `send --apphost-control-plane` 旧 turn loop，也不能作为旧 AppHost control-plane 路径通过口径。
- `send --apphost-control-plane` 已从当前产品路径移除。验收脚本、CLI 工件、线程日志或交付说明中若出现该旧执行路径作为通过证据，本轮验收必须判定为无效。
- 若 CLI 工件中存在 `executionPath` 字段，其值必须为 `kernel-runtime-loop` 或明确的 fail-closed 结果；不得出现旧 AppHost control-plane fallback。当前 `chat` 工件若尚未投影 `executionPath`，应以源码开发入口、脚本命令、事件流、线程日志和无旧路径证据共同证明验收跑在当前架构上。
- 验收通过后，若需要发布或用户级体验确认，才按安装规则刷新用户级三件套；刷新安装产物不能反向替代本案例对当前分支源码/构建产物的验证。

### 3.5 Sub-Agent live 自主触发观察矩阵

以下提示用于最终验收中的独立 live 行为观察矩阵，目标是让真实模型面对多个复杂、多证据域的问题，并记录模型是否自主请求 `spawn_agent`。提示词本体只允许描述问题域与交付要求，不得描述方法；不得包含 agent、子任务、并行、委托、派生、拆分、协作、执行轨道、`spawn_agent` 等会暗示调用方式的词。若这些词出现在提示词本体中，对应场景的 live 观察证据无效。

观察矩阵的通过含义不是“必须观察到 `spawn_agent`”。正确结论分两层：

- 机制验收：工具面、治理、物化、子 run、回流链路必须可确定复验。
- 行为观察：每个 live 场景必须记录 provider tool surface 是否包含 `spawn_agent`、模型实际请求了哪些工具、是否观察到自主 `spawn_agent`。观察到自主触发是产品级强证据；未观察到也是有效行为结论，说明当前任务/模型/tool description 下模型选择了其他处理方式。

本轮正式观察协议必须在执行前冻结：固定 3 个任务、3 个模型协议 cell、每格固定 N 轮，默认 `N=3`，即 `3 tasks x 3 model cells x 3 runs = 27` 次 live 观察。脚本允许通过 `-SubAgentLiveRunsPerCell <N>` 调整 N，但调整后的 N 必须出现在结果 JSON 中；同一次矩阵运行不得因为已观察到或未观察到 `spawn_agent` 而提前停止、追加场景、删除场景或改写判读规则。最终必须同时报告 `spawnObserved/total` 触发率、`effectiveSpawnObserved/total` 有效回流率和 `falsePositiveSpawnObserved/total` 弱信号/误触发率；各 `task x model` cell 也必须分别给出 `TriggerRate`、`EffectiveRate`、`FalsePositiveRate`。`0/N` 与 `N/N` 都是合法工程结论，但只有 `EffectiveRate > 0` 才能作为产品级模型自主 Sub-Agent 强证据。

### 3.6 多 Agent 确定性最终验收案例

本案例还必须包含一条确定性多 Agent 机制门禁，用于证明多 Agent 能力本身已经可用。该门禁由当前源码构建的 `tools/acceptance/TianShu.SubAgentAcceptance` 执行，不依赖真实 provider 的自主选择，因此不得用 3.5 的 live 观察矩阵替代。

该确定性案例必须在同一个 `evidence.json` 中同时观察到：

- 并行 fanout：父模型一次请求至少 3 个 `spawn_agent` 子项，`maxConcurrentAgents >= 2`，且子项执行窗口存在重叠。
- 子树治理：父级允许 `spawn_agent`，子治理经 `SubAgentGovernanceNarrowing` 收窄后不再允许子项继续 `spawn_agent`，且 side effect 与模块权限不超过父边界。
- 预算切分：每个子项必须有独立 `allocatedBudget`，且子项实际 requested budget 不超过分配预算。
- 结果 fan-in：父级必须收到同一 fanout 的 `subAgentFanInSummary`，并把所有子项结果回流到下一次模型调用。
- 失败隔离：至少一个子项以确定性失败码结束，另外至少两个子项成功，父级 fan-in 状态为 `CompletedWithFailures`，父 run 仍能完成。
- 整树复盘：必须生成 `subAgentTreeDiagnostics`，至少包含 1 个根节点、3 个子节点、3 条边、replay/diagnostics 引用和 `sub_agent.tree_diagnostics.v1` schema 标记。

确定性多 Agent 案例是最终验收必过项；live 自主触发观察矩阵是产品行为证据。两者结论必须分开记录：确定性案例失败时，本轮最终验收失败；live 矩阵未观察到自主触发时，只能说明当前任务/模型/tool surface 下未观察到模型自主调用，不能否定机制门禁。

- 已出现至少一次真实工具调用
- 当前回合尚未完成
- 引导消息应插入到合适边界，而不是丢失、覆盖上下文或冲掉上一轮显示内容

对齐 Codex 的期望语义是：

- 在“下一次模型工具调用边界”后插入引导
- 引导消息进入会话后，应真实影响后续执行方向

### 4.2 中途中断

中途中断必须满足以下条件：

- Agent 已经开始工作
- 已出现至少一次真实工具调用
- 当前回合尚未完成
- 用户意图是停止当前回合，而不是排队等待

中断后必须满足：

- 旧回合停止继续追加尾流输出
- 当前轮状态进入已中断或等价终态
- 后续继续/改向消息成为新的主线

- 脚本必须硬性检查 Remote Continuity 机制门禁证据：`deterministic-remote-continuity`、只读 follower 事件、`FromCursor` replay、snapshot refresh、远程审批 accepted、interrupt/resume accepted、follow-up accepted、重复 follow-up `DuplicateIgnored`，并确认 snapshot redaction 记录 `absolute_path`。
- 脚本必须单独记录 live 观察实验是否出现真实模型自主 `spawn_agent`。live 工具面证据以 `runtimeDiagnosticsProjection.providerToolSurface` 为准，每个 run 必须能看到 `hasSpawnAgent=true`，且该证据只记录工具名集合，不保存完整 provider request 或 secret。结果分三层：若矩阵中任一 run 真实触发并完成回流，标为“观察到有效自主触发，产品级自主 Sub-Agent 证据成立”；若只观察到弱 `spawn_agent` 信号但缺少 bridge / toolResults / completed disposition 等有效回流证据，计入 `FalsePositiveSpawnObserved`，不得作为产品级证据；若完整矩阵未触发，标为“机制门禁通过，当前任务/模型/tool surface 下未观察到自主触发”，该结论有效但不得宣称模型自主多 Agent 已通过；若任一计划 run 执行失败、工具面缺失、证据不足或提示词含方法诱导，标为“观察矩阵无效，需要按同一冻结协议重跑”。脚本不得早停，不得只截取有利样本。
- `tools/Test-TianShuV09MultiAgentReleaseGate.ps1` 是 v0.9 发布门禁：它只检查默认不暴露 `spawn_agent`、显式启用必须带 `--approve-all`、确定性多 Agent 机制证据完整，以及本脚本输出的 live 观察报告字段存在。真实 provider 的 live 自主触发结果仍由本最终验收脚本完整执行后公开记录，不能用发布门禁替代 live 观察矩阵。

## 7. 过程观察点

### 7.1 清理动作是否真实发生

- 是否从干净的 `Test\TianShu最终验收` 开始
- 是否清掉了上一轮 CLI 辅助目录 `Test\TianShu最终验收.__cli`
- 是否避免了上一轮项目、导出文件、构建输出和工件结果干扰本轮判断

### 7.2 中途引导是否真正生效

- 引导消息是否在回合中途被正确接收
- 引导后，Agent 是否明显切换到“以 Codex 当前配置契约、官方 schema 和现行实现为准”的实现路径
- 引导消息是否保持在正确会话位置，而不是吞掉、覆盖或冲掉上一轮内容

### 7.3 中途中断并继续/改向是否真正生效

- 中断动作是否真正打断了当前回合
- 中断后，旧回合的工具调用与输出是否停止追加
- 中断后的新消息是否成为后续主线
- 中断后是否保留了必要上下文，而不是丢失已确认有效的信息

### 7.4 Sub-Agent 必验项

Sub-Agent 验收分两层，必须分开记录：

- 机制门禁：当前源码构建的 harness 必须证明 `spawn_agent` 由模型输出进入 `RuntimeComposition`，被物化为 `ModuleCapabilityStep(module.sub_agent/sub_agent.spawn)`，再由 `ExecutionRuntimeSubAgentModuleBridge` 调用真实 `SubAgentOrchestrationModule`，并把子 run 结果作为 `toolResults[]` 回流父模型。
- 多 Agent 确定性门禁：当前源码构建的 harness 必须在同一案例中证明并行 fanout、子树治理、预算切分、结果 fan-in、失败隔离与整树复盘全部成立；这是 v0.9 多 Agent 能力是否可用的必过证据。
- live 自主触发观察：真实 provider 模型面对 3.5 的复杂多源审计任务矩阵时，若任一计划 run 自主调用 `spawn_agent` 且结果回流成功，则本轮可判定“产品级模型自主 Sub-Agent 通过”；若完整矩阵均未调用，则必须记录为“机制门禁通过，当前任务/模型/tool surface 下未观察到自主触发”，这是有效观察结论，不能偷算为通过或失败；若矩阵未完整执行、证据不足或提示词含方法诱导，则本轮 live 观察无效，需要按冻结协议重跑。
- 旧 agent job / 协作树 UI / `send_input` / `wait_agent` / `close_agent` 不是本节通过证据，只能作为历史或未来扩展观察项。

### 7.5 实际交付是否成立

- 是否真的创建了 `.NET 10 WPF` 项目
- 是否真的生成了 GUI，而不是只写说明文档或半成品代码
- 是否真的具备导出 Codex `config.toml` 的能力
- 是否真的进行了构建
- 是否真的尝试拉起界面供人工验收；若自动脚本使用 `-SkipGuiLaunch`，必须由人工另行启动 GUI 并记录结果

### 7.6 最终交付说明是否可审计

- 是否明确列出配置依据文件，而不是只笼统声称“参考了源码”
- 是否列出本轮主要覆盖的配置域，以及未覆盖、降级或取舍的配置域与原因
- 是否列出生成项目路径、构建命令与构建结果
- 是否列出导出 `config.toml` 的路径或操作入口
- 是否列出中途引导、中断恢复、Sub-Agent 机制门禁、Sub-Agent live 自主触发观察状态和 GUI 启动状态的可复核证据

## 9. 任一项命中即判失败

出现以下任一情况，即判定本案例失败：

- 没有先清理上一轮生成物
- 修改了当前用户级 Codex 配置 `%USERPROFILE%\.codex\config.toml`
- 配置项主要依据来自用户配置、TianShu 自身配置、help 文本或过时资料，而非 Codex 当前配置契约、官方 schema 和现行实现
- 只生成了代码片段、说明文档或不可运行半成品
- 没有构建
- 没有尝试拉起界面
- 中途引导消息未送达、未生效、导致上下文错乱，或无法正确插入当前工作流
- 中断未真正打断当前回合，或中断后旧回合尾流仍持续刷新覆盖新方向
- Sub-Agent 机制门禁缺失或失败
- 多 Agent 确定性门禁缺失或未能在同一案例中证明 fanout、子树治理、预算切分、fan-in、失败隔离与整树复盘
- 把旧 agent job / 协作树 UI / 文本提及 `spawn_agent` 误当作当前 Kernel Runtime loop 的 sub-agent 通过证据
- live 观察矩阵未触发真实 `spawn_agent` 却宣称“模型自主 Sub-Agent 已通过”
- live 观察矩阵因为观察到或未观察到某个结果而提前停止、追加/删除样本、改写 N 或只截取有利样本
- live 观察实验提示词含 agent、子任务、并行、委托、派生、拆分、协作、执行轨道、`spawn_agent` 等方法诱导词，却仍把结果作为自主触发证据
- 未检查 CLI 工件与线程日志，仅以退出码宣称通过
- 最终交付说明缺少能力证据清单，或无法说明配置依据、覆盖边界、构建结果、导出能力与关键过程证据
- 最终结果无法支撑人工实际验收
